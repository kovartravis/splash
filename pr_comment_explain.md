At the heart of any agentic development environment is the relationship between the environment itself—which holds the context, files, and state—and the agent process, which performs the reasoning and actions. In Splash, the agent runs as a child process (the "harness") within a terminal emulator. For the agent to actually perceive and manipulate the workspace, it relies on the Model Context Protocol (MCP). The editor runs an embedded MCP server, and the agent acts as an MCP client. 

However, until now, there was a missing link in this architecture. When Splash spawned an agent like `agy` or `claude`, it spun up the process but left the agent entirely oblivious to the embedded MCP server's existence. The user was forced to manually bridge this gap, explicitly configuring the agent to connect to the correct local port. In a seamless, integrated environment, this kind of manual plumbing breaks the illusion of a cohesive system; the environment should configure the agent automatically at boot time.

To solve this, we need an automatic handshake at the process boundary. Since Splash is the parent process responsible for spawning the harness, it already holds the definitive source of truth: the exact dynamic URL of its embedded MCP server. The most robust way to transmit this truth to a child process is at the moment of creation, injecting the configuration directly into the process's startup arguments before it ever begins executing. 

In this iteration, we implemented this automatic discovery mechanism by augmenting the `PtyHarness::spawn` lifecycle in `src/pty.rs`. We intercept the command construction phase and inspect the target executable. If we detect that the user is launching `agy`, we dynamically append the `--mcp-server` flag alongside the server's URL. For `claude`, which consumes its configuration differently, we dynamically synthesize a temporary JSON configuration file on disk specifying the MCP connection details, and then pass this file via the `--mcp-config` flag. 

To guarantee that this process boundary behaves as expected, we codified these invariants through Test-Driven Development. We introduced targeted unit tests that spin up mock shell scripts acting as our agents. By capturing and verifying the exact arguments passed down to the pseudo-terminal layer, we ensure that Splash correctly manipulates the CLI arguments for both agent types without accidentally bleeding configurations where they don't belong.

Ultimately, these changes transform the startup sequence from a manual chore into a zero-configuration experience. When an agent wakes up inside Splash, it is now instantly and automatically wired into the surrounding environment, ready to observe and manipulate the workspace from the very first keystroke.
