At the heart of many modern development tools is the need for decoupled processes to communicate seamlessly. When a terminal multiplexer or layout engine runs a child process—often a specialized "harness" or agent—that child must understand the context in which it operates. Without an explicit channel of communication, the child is blind to the structure of the surrounding user interface, unable to see the tabs, panes, and file trees that the user is interacting with. To bridge this gap, we need a mechanism that allows the host application to expose its internal state to the spawned processes in a standardized, language-agnostic way.

This is precisely the problem the Model Context Protocol (MCP) aims to solve, and integrating it begins with establishing a reliable transport layer. In our system, the most universal way for two local processes to converse is over HTTP. By embedding a lightweight HTTP server directly within the main event loop of our application, we create a listening post. This server binds to a dynamically assigned local TCP port—a crucial detail that prevents port collisions when running multiple instances or concurrent test suites.

However, a listening server is only useful if the child processes know how to reach it. When the application spawns a new pseudoterminal (PTY) for a harness, it now explicitly passes the dynamically chosen port down to the child via an environment variable, `SPLASH_MCP_URL`. This serves as an immediate handshake. From the moment the child process wakes up, it possesses the exact address required to query its parent, eliminating the need for complex discovery protocols or brittle configuration files.

With the communication channel established and discoverable, the application must actually speak the protocol. We introduced a rudimentary yet effective request-handling loop within the application's tick cycle. When the embedded server receives an MCP tool call—specifically the `list_layout` command—it iterates through the application's internal data structures. It translates the deeply nested tree of tabs, vertical splits, horizontal panes, and their respective contents into a clean, easily consumable JSON representation. This allows the child agent to request a real-time snapshot of the user's workspace at any moment.

To ensure this machinery operates correctly under real-world conditions, we also constructed an end-to-end integration test. This test programmatically launches the application, spawns a minimal shell harness, captures the injected environment variable, and issues a live HTTP request back to the embedded server. By asserting that the returned JSON correctly reflects the initial pane configuration, we guarantee that the entire communication loop—from port binding to environment injection to JSON serialization—functions cohesively as a single, unified system.
